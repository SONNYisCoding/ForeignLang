# ForeignLang - Architecture & Database Schema

## Step 2: Database Schema Draft (PostgreSQL)

We will use a relational model to ensure data integrity, especially for subscriptions and content relationships.

### 1. Identity & Subscription
**Table: `users`**
*   `id` (UUID/Long): Primary Key.
*   `email` (Varchar, Unique): From Google OAuth.
*   `full_name` (Varchar): Display name.
*   `role` (Enum/Varchar): 'GUEST', 'LEARNER', 'ADMIN'.
*   `subscription_tier` (Enum): 'FREE', 'PREMIUM'.
*   `created_at` (Timestamp).

**Table: `usage_quotas`**
*   `user_id` (FK -> users.id): One-to-One relationship.
*   `daily_requests_count` (Integer): Current usage today.
*   `last_reset_date` (Date): To track when the counter was last reset.
*   *Logic:* If `date(now()) > last_reset_date`, reset count to 0.

**Table: `subscriptions` (Optional for MVP, good for audit)**
*   `id` (PK).
*   `user_id` (FK).
*   `start_date` (Timestamp).
*   `end_date` (Timestamp).
*   `status` (Active/Expired).

### 2. Dynamic Content (CMS)
**Table: `topics`**
*   `id` (PK).
*   `title` (Varchar): e.g., "Business Email Writing".
*   `description` (Text).
*   `difficulty_level` (Enum): Beginner, Intermediate, Advanced.

**Table: `lessons`**
*   `id` (PK).
*   `topic_id` (FK -> topics.id).
*   `title` (Varchar): e.g., "Formal vs Informal Greetings".
*   `content_body` (Text/JSON): The actual learning material (Markdown supported).
*   `order_index` (Int): To sort lessons within a topic.

**Table: `templates`**
*   `id` (PK).
*   `topic_id` (FK -> topics.id).
*   `name` (Varchar): e.g., "Job Application Template".
*   `structure` (Text): The boilerplate text with placeholders.

### 3. Learning Activity
**Table: `user_submissions`**
*   `id` (PK).
*   `user_id` (FK).
*   `lesson_id` (FK, Optional).
*   `input_text` (Text): What the user wrote.
*   `ai_feedback` (Text/JSON): The stored response from the AI Evaluator.
*   `created_at` (Timestamp).

## Step 3: High-Level Architecture & AI Integration

### 1. Spring Boot Web Architecture
We will follow a standard **Controller-Service-Repository** pattern.

*   **Controller Layer (`@RestController`):**
    *   Handles HTTP Requests (GET, POST).
    *   Validates DTOs (Data Transfer Objects).
    *   *Example:* `LessonController` exposes `GET /api/v1/lessons/{id}`.
*   **Service Layer (`@Service`):**
    *   Contains business logic.
    *   *Example:* `UsageService` checks if a user has enough quota before calling `AiService`.
*   **Repository Layer (`@Repository`):**
    *   Interfaces extending `JpaRepository`.
    *   Handles SQL queries automatically.

### 2. Docker Compose for Local DB
Since your team has mixed OS environments (likely Windows/Mac), Docker Compose is crucial for a consistent database setup.
*   **File:** `compose.yaml` (already generated by Spring Initializr).
*   **Function:** It spins up a PostgreSQL container.
*   **Benefit:** No need for team members to install Postgres manually. They just run `docker-compose up`.

### 3. AI Orchestration Pattern (Strategy Pattern)
To manage different AI personas (Consultant, Teacher, Evaluator), we should avoid a giant `if-else` block.

**Concept:**
Define a common interface `AiPersonaHandler`.

```java
public interface AiPersonaHandler {
    String getSystemPrompt();
    AiResponse generateResponse(UserContext context, String userInput);
    PersonaType getType(); // Enum: CONSULTANT, TEACHER, EVALUATOR
}
```

**Implementations:**
1.  `ConsultantHandler`: Uses a prompt focused on navigation and sales.
2.  `TeacherHandler`: Uses a prompt focused on explaining concepts and being patient.
3.  `EvaluatorHandler`: Uses a prompt focused on strict grammar checking and scoring.

**Factory:**
A `PersonaFactory` injects all handlers and returns the correct one based on the user's request type.
